### 三刷 (7/23/2026) ❌ 同模板不同 slow 语义：27 的 return slow 习惯带进 26，少了 +1

和 27 同天连刷，循环体全对，只错 return。我的代码：

```java
class Solution {
    public int removeDuplicates(int[] nums) {
        int slow = 0;
        for (int fast = 1; fast < nums.length; fast++) {
            if (nums[fast] != nums[slow]) {
                slow++;
                nums[slow] = nums[fast];
            }
        }
        return slow;   // ❌❌ 核心错误：slow 是「最后一个确认元素的下标」，
                       //    长度 = 下标 + 1 → 应该 return slow + 1
    }
}
```

Dry run（`nums = [1,2]`）：fast=1 时 `2 != 1` → slow=1、写入；循环结束
return slow = **1 ❌**（正确长度 2）——静默错误，不崩，只少一个。

**根因**：拿 `nums[slow]` 当比较对象，slow 就被绑架成「最后确认位」——被迫先加后写，slow 是下标，return 必须 +1。这个语义和 27 的「写入位」正相反，同模板连刷时习惯串了台。

**三刷后的解法升级（当天，与 27/80 打通）**：比较对象决定 slow 语义 → 那就换比较对象。**改和 `nums[fast-1]` 比，slow 从此锁定「下一个待写入位」**，[0, slow-1] 是有效区间，写后加、slow 即个数、直接 return——语义分裂的错误失去生长空间。此后 26 只用这一种写法。

**自测点**：不看答案说出 ① slow 的 invariant 一句话 ② 为什么比较对象必须是 nums[fast-1] 而不是 nums[slow] ③ return 为什么不用 +1

### 正确写法

**slow = 下一个待写入位，[0, slow-1] = 有效区间**，与 27/80 同一套 invariant：

```java
class Solution {
    public int removeDuplicates(int[] nums) {
        int slow = 1;                          // nums[0] 必然保留，slow 从 1 起步
        for (int fast = 1; fast < nums.length; fast++) {
            if (nums[fast] != nums[fast - 1]) {  // 和原数组邻居比，永远不碰 nums[slow]
                nums[slow] = nums[fast];       // 合格 → 搬入：写 + 推进，成对
                slow++;
            }
            // 相等 → 什么都不做，不搬即删除
        }
        return slow;                           // 写后加 → slow 是个数，直接 return
    }
}
```

合格条件「该值第 1 次出现」⇔「和前一个不同」，一次比较判定，不需要 count——这是 k=1 的特权；k≥2（LC80）时「第 2 次出现」无法只看相邻一对判断，count 才出场。

### 复杂度分析

**Time: O(n)**：fast 从 1 走到末尾，每轮必进，loop 恰好 n-1 轮，每轮 O(1)。
**Space: O(1)**：in-place 覆盖，只有 slow/fast 两个 variable。

### 沉淀

- **「移除元素」类统一模板**：fast 只管扫描，slow 只管写入（= 下一个写入位，[0, slow-1] 有效区间），判断交给中间的 if；比较参照物用原数组邻居（nums[fast-1]）或外部值（27 的 val），**永远不拿 nums[slow] 当判断依据**。合格 = 写入+推进成对，不合格 = 什么都不做，写后加 → return slow。26 (k=1) / 27 (按值) / 80 (k=2, 加 count) 全是实例
- **比较对象决定 slow 语义**：拿 nums[slow] 参与比较，slow 被绑架成「最后确认位」，return 被迫 +1、加写顺序被迫颠倒——三刷的错就长在这里，统一模板通过换比较对象把这个坑整个填掉
- 和 69/34 的 return 错同族：都是「循环结束后说不清 pointer 语义」→ 靠猜 return → 静默错误

### 关联

- 27（同模板：比较对象是外部值 val，slow 天然是写入位）
- 80（同模板 k=2：多一个 count state；三刷同日一刷，26 的 return 错在 80 一稿同款重现）
- 69 / 34（「退出时 pointer 语义决定 return 谁」同族）
